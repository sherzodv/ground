
# Ground — Full Vision

Everything here is out of scope for the current RFC but should not be lost.
The RFC is focused on getting a single minimal service running on AWS via
Terraform. This document captures where Ground is going beyond that.

## Architecture

```
                              internet
                                  │
              ┌───────────────────┴───────────────────┐
              │                                       │
       ┌──────▼──────┐                         ┌──────▼──────┐
       │   web-app   │                         │      lb     │
       │    (edge)   │────────────────────────►│    (edge)   │
       └─────────────┘                         └──────┬──────┘
        app.example.com           api.example.com     │
                                                ┌──────▼──────┐
                                                │   svc-api   │
                                                └──┬───┬───┬──┘
                                                   │   │   │
                                       ┌───────────┘   │   └───────────┐
                                       │               │               │
                                ┌──────▼──────┐ ┌──────▼──────┐ ┌──────▼──────┐
                                │  svc-core   │ │   svc-pay   │ │  svc-media  │
                                └──────┬──────┘ └──────┬──────┘ └──────┬──────┘
                                       │               │               │
                                       └───────┬───────┘        ┌──────┴──────┐
                                               │                 │             │
                                        ┌──────▼──────┐  ┌──────▼─────┐ ┌────▼────┐
                                        │  postgres   │  │  postgres  │ │   s3    │
                                        │  (shared)   │  │  (media)   │ │ bucket  │
                                        └─────────────┘  └────────────┘ └─────────┘
```

- `web-app` and `lb` are edge entities — public-facing with domains and TLS
- `web-app` is a static frontend served via CDN, routes traffic to `lb`
- `lb` is a load balancer that terminates TLS and routes to `svc-api`
- `svc-core`, `svc-pay`, `svc-media` are internal services, only reachable from `svc-api`
- `svc-core` and `svc-pay` share a postgres instance
- `svc-media` has its own postgres and an S3 bucket for file storage

## Additional Primitives

### `edge`

A public-facing entity with a domain and TLS. Always the boundary between the
internet and internal services. Can be:

- a CDN-backed static build (e.g. web-app served via CloudFront + S3)
- a load balancer that terminates TLS and routes to internal services
- a reverse proxy

```ground
edge web-app {
  build:  web-app:prod
  domain: app.example.com
  access: [lb]
}

edge lb {
  domain: api.example.com
  routes: [svc-api]
}
```

### `postgres`

A managed PostgreSQL instance. Can be declared standalone (explicit mode) or
inline on a service (simple mode).

```ground
postgres pg-shared {}
postgres pg-media {}
```

### `s3`

An object storage bucket.

```ground
s3 media-bucket {}
```

### `redis`

A managed Redis cache.

```ground
redis cache {}
```

### `external`

A third-party or external API endpoint. Declared explicitly to make egress rules
and access control visible in the Ground config rather than hidden in app code.

```ground
external stripe {
  host: api.stripe.com
}
```

## Access

`access` is a universal permission declaration applicable to all resource types.
Ground derives all necessary infrastructure from a single `access` entry:
network rules, IAM roles, credentials, and injected environment variables.

```ground
service svc-pay {
  access: [pg-shared: read, svc-api, stripe]
}
```

Access levels:
- `read` — read-only access
- `write` — full access (includes delete for S3)
- default (unspecified) — `write`

## Dependency Declaration

### Simple mode

Resources declared inline on the service. Ground creates them and grants full
access implicitly. Best for owned, unshared resources.

```ground
service svc-media {
  image:    svc-media:prod
  postgres: pg-media
  s3:       media-bucket
}
```

### Explicit mode

Resources declared independently. Services reference them via `access`. Required
for shared resources or fine-grained permissions.

```ground
postgres pg-shared {}

service svc-core {
  image:  svc-core:prod
  access: [pg-shared]
}

service svc-pay {
  image:  svc-pay:prod
  access: [pg-shared: read]
}
```

Both modes can be mixed freely.

## Multi-provider deploy

The same stack can be deployed to multiple providers. Each `deploy` declaration
is provider-specific; the stack itself stays cloud-agnostic.

```ground
region us-east {
  aws:   us-east-1
  gcp:   us-east1
  azure: eastus

  zone 1 { aws: us-east-1a, gcp: us-east1-a, azure: 1 }
  zone 2 { aws: us-east-1b, gcp: us-east1-b, azure: 2 }
}

deploy to aws {
  stacks: [prod, staging]
}

deploy to gcp {
  stacks: [prod]
}
```

## Infra ownership modes

**Ground owns everything** — describe your services and stacks, Ground generates
the full infrastructure: network, cluster, subnets, and services. No external
inputs. Best for new projects and small teams.

**Ground references existing infra** — large teams often share a VPC, cluster,
or subnet set across many projects managed by a platform team. Ground can
reference those existing entities instead of creating its own.

## Full example

```ground
region us-east {
  aws:   us-east-1
  gcp:   us-east1
  azure: eastus

  zone 1 { aws: us-east-1a, gcp: us-east1-a, azure: 1 }
  zone 2 { aws: us-east-1b, gcp: us-east1-b, azure: 2 }
}

env production {
  LOG_LEVEL: info
}

env staging {
  LOG_LEVEL: debug
}

external stripe {
  host: api.stripe.com
}

postgres pg-shared {}
postgres pg-media {}
s3 media-bucket {}
redis cache {}

edge web-app {
  build:  web-app:prod
  domain: app.example.com
  access: [lb]
}

edge lb {
  domain: api.example.com
  routes: [svc-api]
}

service svc-api {
  image:   svc-api:prod
  scaling: { min: 2, max: 10 }
  access:  [svc-core, svc-pay, svc-media, cache]
}

service svc-core {
  image:  svc-core:prod
  access: [pg-shared]
}

service svc-pay {
  image:  svc-pay:prod
  access: [pg-shared: read, stripe]
}

service svc-media {
  image:  svc-media:prod
  access: [pg-media, media-bucket: write]
}

group backend {
  web-app
  lb
  svc-api
  svc-core
  svc-pay
  svc-media
}

stack prod {
  env:    production
  region: us-east
  zone:   [1, 2]
  group:  backend
}

stack staging {
  env:    staging
  region: us-east
  zone:   [1]
  group:  backend
}

deploy to aws {
  stacks: [prod, staging]
}
```
